## 🔴 Bug：Story 状态前端死代码

**位置：**

```
frontend/src/components/StoryCard.tsx  第 27-30 行
backend/app/services/story_engine.py   story_status() 函数
```

**症状：**

前端的 `statusClass` 判断了 `"breaking"`（红底）和 `"hot"`（琥珀底）两种状态，但后端 `story_status()` 只返回 `"new"` / `"developing"` / `"stable"`。这两个分支永远不会触发，是死代码。

```typescript
// StoryCard.tsx L27-30 — "breaking" 和 "hot" 永远匹配不到
const statusClass =
    story.status === "breaking"      // ❌ 死代码
      ? "bg-red-100 text-red-700"
      : story.status === "hot"       // ❌ 死代码
      ? "bg-amber-100 text-amber-700"
      : story.status === "new"       // ✅ 可用
      ...
```

```python
# story_engine.py story_status() — 只有三种返回值
def story_status(story: models.Story) -> str:
    """new (<2h), developing (<24h), stable otherwise."""
    ...
    if age < timedelta(hours=2):
        return "new"
    if age < timedelta(hours=24):
        return "developing"
    return "stable"
```

**修法（二选一）：**

A. 给后端补判定逻辑 —— 比如 `heat_score >= 50` 为 `"hot"`，首次出现 < 30min 为 `"breaking"`
B. 前端去掉 `breaking` / `hot` 分支，只保留 `new` / `developing` / `stable`

**建议选 A**，因为 README 和视觉设计明显想要这个区分能力。

**验证方法：** 跑后端烟测确认状态值正确，然后前端构建通过。

## 🟡 设计问题：WatchRule 匹配阈值脆弱

**位置：**

```
backend/app/services/matcher.py  第 29 行
```

**症状：**

`matching_stories_for_rule()` 在数据库层取 `limit * 3` 条 Story，再全量拉到 Python 做 `any(keyword in text)` 过滤。关键词如果冷门，可能 300 条里 0 条命中，用户永远看不到匹配结果。

```python
# matcher.py L29 — limit*3 是硬上限，冷门关键词可能全军覆没
stories = db.query(models.Story).options(...).order_by(...).limit(limit * 3).all()
return [s for s in stories if rule_matches_story(rule, s)]  # Python 层过滤
```

**修法：**

- 把关键词匹配推到 SQL 层，至少用 `OR` 拼接 `text LIKE '%keyword%'` 或 SQLite FTS
- 如果保持 Python 过滤，需要自动翻页逻辑：第一页没找够就 `offset += limit*3` 继续
- 同时去掉 `limit * 3` 的魔法数字，改成可配置参数

**验证方法：** 创建冷门关键词规则 → 调用 API，确认能返回匹配结果（即使匹配在较老的 Story 里）。

## 🟡 打磨问题 2：fetcher.py 异常路径代码意图模糊

**位置：**

```
backend/app/services/fetcher.py  except 块
```

**症状：** `db.rollback()` 后再 `db.commit()` + `db.refresh(log)` 虽然能过（log 已有 id），但读代码时不直观。

**修法：** 加注释说明「log 在 rollback 前已 commit 且有 id，异常路径只更新 log 状态」，或者在 rollback 后不开新 commit 而是直接把状态写入新 session。不改也能正常工作，只是可读性优化。
